# Latticework: Reframe

One-shot. The single most valuable move in judgment is noticing you are solving
the wrong problem. This skill exists to force that check.

1. **The stated problem.** Repeat back, in one line, the question or problem
   the user is working on — as they framed it.
2. **The hidden frame.** Name what the current framing assumes: what it takes
   as fixed, what it optimizes for, whose perspective it uses, what it leaves
   out. Every frame is a choice, usually an unconscious one.
3. **Alternative framings.** Offer 3–4 genuinely different ways to see the
   same situation. Each must:
   - State the reframed question in one line
   - Name what shifts (the constraint relaxed, the variable changed, the
     stakeholder swapped, the time horizon moved)
   - Say concretely how the answer changes under this frame
   Not variations on a theme — each should make the user see something they
   were not seeing before.
4. **The sharpest reframe.** Pick the one most likely to change what the user
   actually does, and say why.

Pull from these moves when generating frames (don't list them — use them):
- Flip the constraint: what if the thing treated as fixed is actually variable?
- Change the level: zoom in (what's the real bottleneck?) or zoom out (what
  game is this actually part of?)
- Swap the stakeholder: whose problem is this really? Who else is affected?
- Shift the time horizon: does this look different at 1 week vs 1 year vs 5 years?
- Invert the goal: what if you optimized for the opposite?
- Remove the problem: what if you didn't have to solve this at all?

A reframe that doesn't change any downstream action was not a reframe.
